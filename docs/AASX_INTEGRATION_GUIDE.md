# AASX Integration Guide

## Overview

The **AASX Integration** feature in the Digital Twin Registry automatically discovers and registers digital twins from processed AASX files. This feature bridges the gap between the ETL pipeline and the twin registry, enabling seamless digital twin management.

## What is AASX Integration?

AASX (Asset Administration Shell Package) files contain digital twin data that has been processed by the ETL pipeline. The AASX Integration feature:

- **Discovers** processed AASX files from the ETL output
- **Auto-registers** digital twins based on AASX data
- **Tracks** sync status between twins and their source AASX files
- **Organizes** data by projects and files
- **Shows** available output formats for each file

## How to Use AASX Integration

### Step 1: Access the Feature

1. Navigate to the **Digital Twin Registry** page
2. Click the **"AASX Integration"** button in the header
3. The AASX Integration section will appear below the main twin registry

### Step 2: Load Projects

1. Click **"Load Projects"** to discover all available projects
2. The system will scan the ETL output directory and display:
   - Project IDs
   - Project names (shortened for readability)
   - Number of files in each project
   - Creation dates
   - Legacy project indicators (if applicable)

### Step 3: View Project Files

1. **Option A**: Select a project from the dropdown and click **"Auto-Register Project"**
2. **Option B**: Click **"View Files"** next to any project to see its AASX files
3. **Option C**: Click **"Discover Files"** to see all processed AASX files across all projects

### Step 4: Understand the File Information

Each AASX file shows:

| Column | Description |
|--------|-------------|
| **File Name** | The original AASX filename |
| **Project** | The project ID this file belongs to |
| **Processed At** | When the ETL pipeline processed this file |
| **Twin Name** | Auto-generated name for the digital twin |
| **Type** | Twin type (extracted from AASX data) |
| **Available Formats** | Output formats generated by ETL (JSON, YAML, CSV, Graph, RAG, Vector DB, SQLite) |
| **Status** | Whether a twin is registered for this file |
| **Actions** | Register twin or view sync status |

### Step 5: Register Digital Twins

#### Individual Registration
1. Find an AASX file with "Not Registered" status
2. Click **"Register Twin"** next to the file
3. The system will:
   - Extract twin information from the AASX data
   - Generate a unique twin ID
   - Register the twin in the database
   - Show success confirmation

#### Bulk Registration
1. Select a project from the dropdown
2. Click **"Auto-Register Project"**
3. The system will register twins for all files in that project
4. View results showing successful and failed registrations

### Step 6: Monitor Sync Status

1. For registered twins, click **"Sync Status"**
2. View:
   - Last sync timestamp
   - Number of data points
   - Recent sync history
   - Current sync status

## Understanding the Data Flow

```
ETL Pipeline → Project-Based Output → AASX Integration → Twin Registry
     ↓              ↓                        ↓              ↓
AASX Files → Organized by Project → Discover & Register → Digital Twins
```

### Project Organization

The ETL pipeline organizes output by:
```
output/projects/
├── {project_id_1}/
│   ├── {filename_1}/
│   │   ├── aasx_data.json
│   │   ├── aasx_data.yaml
│   │   ├── aasx_data.csv
│   │   ├── aasx_data_graph.json
│   │   ├── aasx_data_rag.json
│   │   ├── aasx_data_vector_db.json
│   │   └── aasx_data.db
│   └── {filename_2}/
│       └── ...
└── {project_id_2}/
    └── ...
```

### Available Formats

Each processed AASX file generates multiple output formats:

| Format | File | Purpose |
|--------|------|---------|
| **JSON** | `aasx_data.json` | General data exchange and APIs |
| **YAML** | `aasx_data.yaml` | Human-readable configuration |
| **CSV** | `aasx_data.csv` | Spreadsheet analysis and reporting |
| **Graph** | `aasx_data_graph.json` | Neo4j knowledge graph import |
| **RAG** | `aasx_data_rag.json` | AI/ML training and semantic search |
| **Vector DB** | `aasx_data_vector_db.json` | Qdrant vector database metadata |
| **SQLite** | `aasx_data.db` | Structured database queries |

## API Endpoints

The AASX Integration provides several API endpoints:

### Get All Projects
```
GET /twin-registry/api/aasx/projects
```
Returns all available projects with file counts.

### Get Project Files
```
GET /twin-registry/api/aasx/projects/{project_id}/files
```
Returns all AASX files for a specific project.

### Discover All Files
```
GET /twin-registry/api/aasx/discover
```
Returns all processed AASX files across all projects.

### Auto-Register Project
```
POST /twin-registry/api/aasx/projects/{project_id}/auto-register
```
Registers twins for all files in a project.

### Register Individual Twin
```
POST /twin-registry/api/twins/register-from-aasx
```
Registers a single twin from an AASX file.

### Get Sync Status
```
GET /twin-registry/api/twins/{twin_id}/sync-status
```
Returns sync status for a specific twin.

## Database Schema

The AASX Integration uses a SQLite database (`data/twin_registry.db`) with these tables:

### twin_aasx_relationships
Stores the relationship between twins and AASX files:

| Column | Type | Description |
|--------|------|-------------|
| `twin_id` | TEXT | Unique twin identifier |
| `aasx_filename` | TEXT | Original AASX filename |
| `project_id` | TEXT | Project identifier |
| `aas_id` | TEXT | Asset Administration Shell ID |
| `twin_name` | TEXT | Human-readable twin name |
| `twin_type` | TEXT | Type of digital twin |
| `status` | TEXT | Current status (active, pending_sync, etc.) |
| `created_at` | TIMESTAMP | Registration timestamp |
| `last_sync` | TIMESTAMP | Last synchronization time |
| `data_points` | INTEGER | Number of data points |
| `metadata` | TEXT | JSON metadata |

### sync_history
Tracks synchronization history:

| Column | Type | Description |
|--------|------|-------------|
| `twin_id` | TEXT | Reference to twin |
| `sync_type` | TEXT | Type of sync operation |
| `sync_status` | TEXT | Success/failure status |
| `sync_timestamp` | TIMESTAMP | When sync occurred |
| `details` | TEXT | JSON details of sync |

## Troubleshooting

### No Projects Found
- **Cause**: ETL pipeline hasn't processed any AASX files yet
- **Solution**: Run the ETL pipeline first to process AASX files

### No Files in Project
- **Cause**: Project directory exists but contains no processed files
- **Solution**: Check if ETL processing completed successfully

### Registration Fails
- **Cause**: SQLite database file missing or corrupted
- **Solution**: Check file permissions and database integrity

### Sync Status Errors
- **Cause**: Twin not found or database connection issues
- **Solution**: Verify twin exists and database is accessible

## Best Practices

1. **Run ETL First**: Always process AASX files through the ETL pipeline before using AASX Integration
2. **Check File Formats**: Ensure all required output formats are enabled in ETL configuration
3. **Monitor Sync Status**: Regularly check sync status for active twins
4. **Use Project Organization**: Leverage project-based organization for better data management
5. **Backup Database**: Regularly backup the twin registry database

## Example Workflow

1. **Upload AASX files** to the ETL pipeline
2. **Process files** through the ETL pipeline
3. **Navigate** to Digital Twin Registry
4. **Click "AASX Integration"**
5. **Load projects** to see available data
6. **Select a project** and view its files
7. **Register twins** individually or in bulk
8. **Monitor sync status** for registered twins
9. **Use the twins** in other parts of the system

## Integration with Other Components

The AASX Integration works with:

- **ETL Pipeline**: Discovers processed files
- **Knowledge Graph**: Uses graph format data
- **AI/RAG System**: Uses RAG and vector DB formats
- **Twin Registry**: Manages twin lifecycle
- **Quality Analytics**: Uses structured data formats

This integration ensures that digital twins are automatically created and managed based on the latest AASX data, providing a seamless workflow from data processing to twin management.
